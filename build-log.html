Polish · Empty states (Quiet Room)

Seven empty-state surfaces across the app get unified under one treatment system. Before this round each surface had its own ad-hoc copy, emoji, and markup: the feed empty was a newspaper emoji and h3, the People tab empty was a plain emoji-cluster nudge, the All tab empty was an unstyled p, the Showing Up and Crickets sections rendered "Everyone's showing up!" cheerleader copy even when the right answer was silence, and the diary blank page used a flat Caveat line with no visual suggestion that the page was waiting to be written on. The new system introduces the Quiet Room concept: every empty state is a room that's been set up and is waiting, not broken, not sad, not missing data. Three treatment tiers and one diary-context variant cover every surface, all driven by existing Mixtape tokens so both themes auto-resolve.

The Quiet Room, three layers
New css/ui.css section defines .quiet-room (container), .qr-headline (Fraunces italic 500, 1.15rem, var(--ink-soft)), .qr-sub (Manrope 0.78rem, var(--ink-faint)), and .qr-action-frame (1.5px dashed var(--hairline), 10px radius) with a pill button. The container gets two pseudo-elements: ::before is the motif backdrop at 0.35 opacity on a 70s drift animation, ::after is a warmth glow of radial gradients on a 55s drift. Both reuse the drift-motifs and drift-warmth keyframes already defined on .ambient in base.css so there's no new animation code. A new --empty-motif-svg custom property is declared on :root and [data-theme="side-b"] with the two Mixtape motif SVGs (red/blue/yellow for Side A, salmon/teal/yellow for Side B) so the backdrop auto-swaps per theme without any selector overrides on the .quiet-room rules themselves. prefers-reduced-motion disables all three drift animations.
spec
Three tiers plus diary context
Not every empty state gets the full treatment. Full Quiet Room (motif + glow + Fraunces headline + optional dashed action frame) goes on big standalone surfaces: feed empty, People tab empty when the user has zero follows, All tab empty when search returns nothing or the month has no trackers. Nudge card (dashed border + 0.2-opacity motif + compact CTA, class .nudge-card) handles persistent CTAs like the Browse All card at the bottom of the People sidebar. Inline empty (class .inline-empty, just a Fraunces italic line in var(--ink-faint)) covers small section empties so they recede into the view without competing for attention. Diary context is a special variant that wraps the blank-page content in .quiet-room but with border-radius 0 and a Caveat handwriting headline instead of Fraunces, plus three faint ruled-line divs at 0.5 opacity that suggest a notebook page waiting to be written on.
Seven surfaces migrated
js/following-feed.js renderEmptyState drops the newspaper emoji and h3, renders a Full Quiet Room with "nothing new. everyone's quiet today." and a dashed frame containing Browse All plus the hint "see who else is showing up" (hidden on mobile via media query). js/following-people.js: the allEmpty branch early-returns a Full Quiet Room inside a fresh fw-container so the 3:2 grid doesn't constrain it to the left column; the Pinned section gains an else branch (when the user has follows but none pinned) that renders the 📌 Pinned label plus inline-empty "pin someone to keep them here."; the Showing Up empty is now context-aware ("all pinned." when pinnedActive.length is positive, else "no one else showing up this month."); the Crickets section behavior inverts from "render the section with Everyone's showing up! copy" to "don't render the section at all when count is zero," because the absence of a Crickets block is itself the message. js/tracker-all.js renderAllList replaces the plain p.empty-state with a Full Quiet Room carrying two copy variants (search: "nothing matching that." / "try a different name or activity."; no-trackers: "no other trackers yet for this month." / "give it a few days."), and the clearRenderedResults selector picks up the new .all-tab-empty marker class so re-renders purge cleanly. js/tracker-diary.js selectDay else branch wraps the blank page in a quiet-room, adds three diary-empty-line ruled divs, and swaps the write button from a solid Sora-font button to a dashed .diary-write-btn with lowercase "write something."
Copy conventions across every surface
All empty-state copy follows the house voice from MIXTAPE_SPEC §2 and TOAST_SPEC §6: intentionally lowercase for intimacy, period at the end (diary gets an ellipsis "this page is blank..." as the one exception), no em-dashes, no exclamation points, no emoji in headlines since the motif backdrop replaces emoji as the visual element, no startup-cheerful "Oops!" or "Whoops!" tone, specific rather than generic. The Browse Nudge card, always visible at the bottom of the People sidebar even when you have follows, rewrites from "Find people to follow" / "Head to the All tab to discover who else is showing up." to "find more people to follow." / "see who else is showing up this month." with the same dashed-border + motif treatment as the other Quiet Rooms but scaled down (0.92rem headline vs 1.15rem).

Polish · Theme-aware scrollbar

A tiny round that closes a single visual bug: the page scrollbar was a raw browser-default grey bar on every platform, which reads fine on Side A's cream paper but breaks atmosphere on Side B's deep maroon. html has overflow-y: scroll forced on (so the scrollbar gutter is always reserved and switching views never causes a horizontal layout jump), but zero custom styling was layered on top. This round adds the styling, resolved entirely from existing Mixtape palette tokens so both themes auto-swap with no data-theme overrides. No new tokens. One selector block for WebKit/Blink, two new properties on the existing html rule for Firefox, done.

Token mapping, three parts, zero new custom properties
Track resolves to var(--paper-edge), which is #DCCFA6 warm tan on Side A and #3A1D20 deep maroon on Side B. Thumb resolves to var(--ink-faint): #9C8874 muted brown on Side A, #6E504B muted rose on Side B. Thumb hover resolves to var(--ink-soft): #5A463A dark brown on Side A, #B18A84 dusty pink on Side B. Every pairing is intentionally low-contrast so the scrollbar recedes rather than competes. Because all three tokens already have per-theme overrides from the Mixtape palette, the scrollbar follows whatever side the app is on with no extra CSS.
spec
WebKit pseudo-elements with a floating thumb
::-webkit-scrollbar is 8px wide, thin enough to be discreet and wide enough to grab on a trackpad or touchscreen. ::-webkit-scrollbar-track paints the track background. ::-webkit-scrollbar-thumb uses a fully rounded 9999px radius plus a 2px solid transparent border and background-clip: padding-box, which is the standard trick for making the thumb appear to float inside the track with a small air-gap on both sides instead of filling edge-to-edge. ::-webkit-scrollbar-thumb:hover swaps to the hover token so the thumb darkens slightly when you grab it. Covers Chrome, Safari, and the current Blink-based Edge.
Firefox via scrollbar-width + scrollbar-color
Firefox doesn't support ::-webkit-scrollbar pseudo-elements, so it uses a different mechanism: scrollbar-width (auto, thin, none) and scrollbar-color (two colors, thumb first, track second). Both properties land on the existing html rule in base.css alongside overflow-y: scroll. scrollbar-width: thin matches the 8px WebKit sizing close enough; scrollbar-color: var(--ink-faint) var(--paper-edge) wires into the same tokens. Firefox doesn't support hover states or border-radius on scrollbar thumbs via these properties, which is fine; the spec explicitly accepts the reduced treatment rather than trying to ship a polyfill.
No data-theme override needed
This is the point of the round. Because every part of the scrollbar resolves to a token that already auto-swaps via [data-theme="side-b"] in variables.css, the scrollbar rules themselves can stay theme-agnostic. One block of CSS handles both themes. Toggling between Side A and Side B recolors the scrollbar instantly with no flash, no layout shift, and no additional selectors. A minor detail that would have been three selectors' worth of duplication without the token discipline established in the earlier Mixtape passes.

Polish · Toast system

Small-surface-area round that closes two long-standing items in one pass. The toast system grows a proper four-role hierarchy (neutral, info, success, error) with neutral as the new default, fixing a long-running misuse where every quiet confirmation ("coming soon.", "signed out.", "Pinned!") was falling through to the success treatment because success was the default type. Success becomes reserved for genuine celebrations, which the app doesn't fire yet. Every role class is rewritten to read from existing Mixtape tokens so both themes auto-resolve; no new custom properties were added. In the same round, the legacy --color-teal #80B9B9 alias (sitting in variables.css since the pre-Mixtape era) gets fully retired. PO01 started this cleanup with the login dot; this round sweeps the six remaining consumers across following.css and feed-view.css onto var(--accent-support) and removes the alias definition.

Four roles, neutral as the workhorse
The toast system used to offer three types (success, error, info) but had no neutral default, so showToast("message") with no type argument was going to success. Most toasts are not celebrations. The new hierarchy is neutral (quiet confirmations: "saved.", "signed out.", "copied."), info (contextual notices that need slight visual distinction: "coming soon.", "link copied to clipboard."), success (genuine milestone moments: streak hits, first-entry-ever, comebacks, none of which fire yet), and error (failures, "couldn't save. try again?"). The key decision is that success is rare and reserved. Overusing it dilutes its meaning, so the spec explicitly lists what should and shouldn't trigger it.
spec
Role classes, theme-aware tokens
All four role classes in css/ui.css resolve to existing tokens; no new custom properties were needed. .toast-neutral uses var(--paper-deep) bg, var(--ink) text, 1px var(--hairline) border. .toast-info shares the neutral body plus a 3px left accent in var(--blue) and an asymmetric pill radius (10px 99px 99px 10px) so the accent reads as a visual anchor rather than a ring. Side B auto-remaps --blue to the Mixtape teal #5EAAA8 via the existing theme override, so info looks correct on both themes without extra logic. .toast-success uses var(--green) bg with var(--paper) text, which gives cream-on-forest on Side A and dark-maroon-on-teal on Side B automatically (both correct contrast since --green and --paper both flip per theme). .toast-error keeps white text on Side A red but gets a [data-theme="side-b"] override to var(--paper) so the salmon red on Side B reads against dark text. Three candidates were explored in the mockup for success (red, green, yellow); green won because red collides with error and yellow conflicts with a future warning tier.
The one-line default flip
The correctness-fixing edit in js/ui.js is literally one character: showToast's default type parameter changes from "success" to "neutral". Every call site that currently omits the second argument (nine across app.js, diary-strip.js, feed-card.js, following-people.js, privacy-settings.js) auto-corrects in place. "coming soon.", "Pinned!", "Unpinned.", "All unpinned.", "Unfollowed X.", "Privacy updated." all move from green-success to quiet-neutral, which is what they should have been all along. Audit of the full codebase confirmed no existing call site needs its explicit type argument changed; every "info" and "error" remains correct. Three "info" call sites read slightly more like neutral per the tone table ("Diary entry saved.", "Activities updated.", "Stats copied!") but stay on info for this round since they're a copy decision, not a bug.
--color-teal retirement, full sweep
The legacy --color-teal: #80B9B9 hex alias has been sitting in variables.css as a pre-Mixtape holdover since before the Side A / Side B split. PO01 introduced --accent-support (which maps to var(--blue) and auto-remaps to #5EAAA8 on Side B) as the theme-correct replacement, and migrated the login dot. This round finishes the job. Six remaining consumers migrated to var(--accent-support): in following.css, .fsd-pin-card--lowkey border, .fsd-pin-link.cal:hover color, and .fw-tier-sharing (body, text, and border, with the two hardcoded rgba(128,185,185,X) values rewritten as color-mix(in srgb, var(--accent-support) N%, transparent) to match the house approach from PO03's diary tokenization). In feed-view.css, .fw-feed-evt--lowkey border, .fw-feed-evt--lowkey .fw-feed-evt-nudge color, and .fw-feed-evt-link--cal color. The --color-teal: #80B9B9 line is then removed from variables.css; a final grep confirms zero remaining references in css/ or js/. Visual effect: Side B stays in the teal family (the old #80B9B9 was an approximation; --accent-support resolves to the Mixtape-correct #5EAAA8 on Side B). Side A shifts the .cal link, .lowkey tier accent, and .fw-tier-sharing badge from the stale pre-Mixtape teal to Mixtape navy #4F6C8E, which is the intended behavior for a theme-correct cool accent on cream paper.
What's next
Settings entry point is overdue: edit display name, change default diary cover, manage entitlements, toggle theme. Scrollbar polish, empty states with Mixtape flourish, modal audit follow. On-deck readability item from PO04 (no-full-weeks calendar preview copy washing out on Side A) still waiting for a dedicated pass. Icons and landing page stay deferred. Warning toast tier (fifth role, likely yellow) is logged in the toast spec's future considerations but deliberately not pre-built; it waits for a concrete trigger like a destructive-confirmation flow.

Three covers, retuned as registers
The legacy keys coral / cream / indigo are renamed to bold / quiet / moody, and the visuals get reauthored as emotional registers rather than color swaps. DEFAULT_DIARY_COVER = "bold". A mapLegacyCoverKey shim bridged reads during the cutover and was removed at the end of the round after production data migrated.
spec
Dotted paper
All four diary paper surfaces (desktop left page, desktop right page, mobile card, mobile sheet) convert from repeating-linear-gradient rule lines to a radial-gradient dot grid. Dot spacing, dot size, and dot color are tuned per surface so the mobile card's smaller scale stays readable without looking busy. The gesture references Ryder Carroll's bullet journal method: a grid that suggests structure without imposing it, so writing, sketching, and sticker placement all coexist. Popover cover picker
The three sites that used to show a row of colored dots (closed notebook, open modal left page, mobile sheet) converge on a shared picker module. New file js/diary-picker.js exports renderMiniCover (renders a scaled-down cover with the "diary." title), renderCoverPopover (floating popover with thumbnails, captions, selected ring, locked overlay), and renderCoverRow (the same row inline, for mobile where the sheet top bar is the surface). Setup flow restructure, handoff
setup.html becomes a 2-step state machine: Step 1 (display name) and Step 2 (cover picker), with step pips, a back button, and history.pushState so browser back works. Step 2 renders the cover grid via renderMiniCover at 140x180 so users see the real cover art at decision scale, with captions in DM Mono. Continue on Step 2 calls seedEntitlementsIfMissing(uid) before writing diaryCover, which also gates the first entitlements doc creation on the same action. The handoff screen takes the full viewport, sparkle SVG (4-point path, decorative circles stripped) pulses red, then routes to the monthly setup modal. No data field changes on the flow itself; only the shape and the field write (diaryCover, not diaryTheme).
Entitlements architecture
New Firestore collection entitlements/{userId} with fields ownedCovers: string[] and ownedAppThemes: string[]. Security rules allow read and create where request.auth.uid == userId, and deny update and delete outright, so no client can mutate entitlements once seeded (future paid purchases will go through a server-side callable with admin privilege). js/entitlements.js exports DEFAULT_OWNED_COVERS = ["bold", "quiet", "moody"], DEFAULT_OWNED_APP_THEMES = ["side-a", "side-b"], getOwnedCovers(uid) with a safe fallback to defaults, and seedEntitlementsIfMissing(uid) which reads first and creates the doc only if missing. No paid cover ships in this round; the architecture is deliberately ahead of the feature so the paid slice is purely additive.
Per-month cover override
The picker now writes per-month, not to the user default. Path is logs/{yearMonth}/entries/{uid}.diaryCover, merged into the existing monthly entry doc. Reads use getActiveCover(monthCover, userDefaultCover) which returns the month-level value if set, otherwise the user default, otherwise null. Every render site (desktop closed notebook, modal, mobile card, mobile sheet, tracker-mylog preview) resolves the active cover through a Promise.all(getDiaryCover, getMonthCover) pair. The popover shows "just for April" in Fraunces italic below the selected thumbnail when the active cover diverges from the user default, so the per-month state is visible at a glance. User default only changes during setup; future Settings panel will expose a way to retune it.
Admin migration and legacy cleanup
Two admin-gated Cloud Functions on the existing requireAdminAuth + dry-run-default pattern. adminMigrateDiaryCovers scans users/{uid}, maps the legacy diaryTheme value (coral / cream / indigo) to the new key (bold / quiet / moody), writes diaryCover, and leaves diaryTheme intact so the client keeps working across the cutover. adminBackfillEntitlements seeds the default entitlements doc for every user who doesn't already have one. Batches are 400 operations via commitInChunks. Ran in production: 4 users migrated, 40 entitlements seeded (one user already had a doc from the setup slice). adminCleanupLegacyDiaryTheme followed a few minutes later, deleting diaryTheme via FieldValue.delete() for users who have diaryCover populated; skips any user with a legacy value but no new field as a safety rail against out-of-order runs. 4 users cleaned, idempotency confirmed on a second dry-run. Client-side, getDiaryTheme / saveDiaryTheme are now getDiaryCover / saveDiaryCover across five call sites, and the legacy-key fallback in getMonthCover is gone.
What's next
First paid cover is the most natural next slice now that entitlements is in place, but it's a content and checkout scope, not a polish one, so probably not the immediate next round. Settings entry point is overdue: edit display name, change default cover, manage entitlements, toggle theme. Badge palette, per-theme

spec
Markers, reordered with Mixtape sparkle
MARKERS reordered from square / circle / star / heart / check / x / scribble to square / star / circle / heart / check / x / scribble. Square moves to slot 1 because the red filled square is the visual echo of the wordmark period — the most Mixtape-specific shape in the set.
