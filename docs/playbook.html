01

Philosophy

the why, on one page

The aesthetic

Mixtape. Warm 90s and early-2000s nostalgia, not literal. Mid-century book meets cassette tape. The only "mixtape" vocabulary that reaches the UI is "Side A" and "Side B" in the theme picker. Everything else is warm editorial restraint.

The product posture

showup. is 95% personal, 5% social. The tracker and diary are the product. Following and All tabs are supporting features, not the center of gravity. Every design decision respects this ratio.

showup. is premium-first, monetization-minded. The design quality sets a bar that justifies paid themes and future paid tiers. The app should feel worth paying for before we charge anyone.

The five pillars

Every component, every copy string, every animation answers to these five.

  1. 01
    The tracker is the heartbeat. It is what brings people back daily. Never obscure it, never bury it, never slow it down.
  2. 02
    The diary is the emotional peak. It is where the product becomes personal. Protect its intimacy.
  3. 03
    Warm analog, crisp digital. Paper and ink on the surface, precision and speed underneath. Both matter.
  4. 04
    Quiet by default, personality on contact. The app does not shout. It reveals itself as you use it.
  5. 05
    Specific, not generic. Copy uses the person's name. Activities are named inline. Signals reference the actual moment.

Side A / Side B

Side A is the default light theme. Cream paper, deep ink, confident red. Friendly, legible, welcoming. Daytime showup.

Side B is the default dark theme. Maroon leather, warm cream text, salmon red, teal accent. Quiet, contemplative, evening. showup. in the evening.

Side B is not "Side A with the lights off." It has its own emotional register. Together they form one cassette with two sides.

02

Tone & Voice

how showup. sounds

showup. sounds like a quietly observant friend who pays attention. Not a coach, not a cheerleader, not a life-hack blog.

The five traits

  • Warm. Never cold, never clinical.
  • Specific. Uses real names, real activities, real moments.
  • Quietly observant. Notices things without announcing it.
  • A little sarcastic, sometimes. Only when the moment earns it. Never mean.
  • Never performative. No fake enthusiasm, no corporate cheerfulness.

Voice by context

ContextCalibration
Streak hit / milestoneCelebratory but grounded. "day 25. still showing up."
Low key signal copyObservational, third-person-ish, warm. Uses first name.
Ghost / absenceThe feed itself speaking. No person named. Poetic restraint.
Diary copyPersonal, intimate, lowercase where appropriate.
Error messagesDirect, kind, specific. Never alarming.
Confirm dialogsOne word or two. "Sure?" not "Are you sure you want to do this?"
Empty statesPatient, gentle. Never "no data found."

Do's and Don'ts

Instead ofSay

Error: Failed to save dataCouldn't save. Try again.
Are you sure you want to unfollow?Unfollow? Sure?
User has disabled sharingGone quiet for now.

Intentionally lowercase

showup. lowercases certain words for intimacy and texture. These are deliberate, not careless.

  • diary. (the feature name and section label)
  • crickets (the empty section in People view)
  • no tracker this month
  • Toasts: saved. synced. couldn't save.

Sentence case stays for headings, proper nouns, button labels, and anywhere formality serves clarity.

Forbidden patterns

  • Em-dashes. Anywhere. Ever. Use a comma, period, semicolon, colon, or parentheses. This is a hard rule.
  • Exclamation points unless the moment is genuinely celebratory.
  • "The user" in any user-facing string. Say you or use their first name.
  • AI-ish phrasing. "I'd be happy to help." "Let me know if..." This is a product, not an assistant.
  • Startup cheerfulness. "Oops!" "Whoops!" If something failed, say what failed. With kindness.
  • Generic placeholders. "your activities" when we could say the actual names.
Copy review checklist. Before shipping any user-facing string, run it through these: (1) Would a friend say this? (2) Is it specific to showing up, or generic? (3) Does it use the person's name if it should? (4) Does it respect the 95% personal / 5% social ratio? (5) Is anything here an em-dash? (6) Does it need an exclamation point, or is that fake enthusiasm?
03

Hard Rules

non-negotiable

These cannot be overridden by any design decision, feature request, or user preference. Breaking a hard rule breaks the system.

Design system

  • Red is the hero across every theme. Not blue, not green, not purple. The red hue may shift per theme (Side A #C3342B, Side B #F07A5E), but red is always the signature.
  • The stripe order is always red / yellow / [accent], left to right. The accent may vary per theme; the first two never move.
  • Sora 800 is the logo font. Everywhere. At every size. No exceptions.
  • Only "up" in the wordmark gets the red highlight. The period stays in normal text color.
  • Active nav state is always the full ink pill with red track number. Not an underline, not a border, not a color swap on the label.
  • Level 3 texture is the texture. Grain + drifting warmth glow + drifting motif layer. Don't introduce new noise patterns.

Privacy

Cross-referenced from FOLLOWING_SPEC.md. Three rules that never break:

  1. A follower sees the same card in Following tab and All tab. Always identical.
  2. Low key and Ghost are audience-agnostic. Same treatment for followers and strangers.
  3. Private cascades both Calendar and Diary. Always.

Copy

  • No em-dashes. Anywhere. Ever.
  • No exclamation points unless the moment is genuinely celebratory.
  • Never "the user" in any UI string.
  • Never AI-ish phrasing. The app is an app, not an assistant.

Engineering

  • Modularize aggressively. No file bloats past reasonable size.
  • Design tokens live in css/variables.css. Never hardcode hex values in component CSS.
  • Shared modules are single-source-of-truth. If a utility exists, use it.

Motion Language

hover the cards to see motion

Fast enough to feel snappy, slow enough to feel considered. showup. uses one spring curve for almost everything: cubic-bezier(0.22, 1, 0.36, 1). iOS-feel. Crossfades for content swaps, never slides on variable-height panels.

Responsive System

breakpoints & rules

Breakpoints

NameRangeContext
Mobile0 – 767pxPhones, bottom-nav app shell
Tablet768 – 1023pxRare. Treated as desktop with tighter spacing.
Desktop1024px +Full V3 sidebar layout

What scales

  • Typography: display sizes scale down 20-25% on mobile. Body stays 16px.
  • Paddings and gaps: halve on mobile (32 → 16, 24 → 12).
  • Texture: motif scale drops on mobile (260px tiles) to keep motifs visible.

What changes per viewport

  • Nav: desktop V3 sidebar. Mobile bottom nav + header.
  • Month nav: desktop has a dedicated bar. Mobile inlines into the header row.
  • Tracker layout: desktop shows rows per activity. Mobile shows a 7-col month grid.
  • Pinned panel: desktop on the right. Mobile at the top of the feed.

What stays constant

Palette, typography stack, stripe colors, motion principles, and voice are identical at every viewport.

Theme System

two defaults · seven paid candidates

Every theme is a set of CSS custom properties. The theme switcher changes data-theme on <html>, which triggers a different token layer. Components always read from tokens, never hardcode hex values.

A theme change is a token swap, nothing more. If a component references a token correctly, it will theme correctly with zero extra code.

Paid themes: the reserve

Every palette we explored but did not ship as default becomes future Shop inventory. Nothing wasted.

Authoring rules for new themes

  • Red stays the hero. Hue may shift per theme.
  • Stripe order is always red / yellow / [accent].
  • Minimum three-color palette. Red, yellow, accent required.
  • Body text must pass WCAG AA against background.
  • Motifs tuned per theme. Don't reuse Side A's SVG on a dark theme.
12

Iconography

out of scope

Iconography is deliberately out of scope for this document. It will be documented in a future ICON_SPEC.md covering stroke weight, corner radius, fill rules, currentColor inheritance, and brand motifs.

Until ICON_SPEC.md exists, icons in the live app continue their current conventions. The Mixtape redesign does not change icon shapes, only their tint (via currentColor).

Migration Plan

how the redesign actually ships

The Mixtape redesign is a token and structure migration, not a rewrite. Existing features keep their logic and data paths. What changes:

File-by-file order

  1. css/variables.css Add Side B token layer, paid-theme placeholder layer
  2. css/base.css Load Fraunces, Manrope, DM Mono, Caveat via Google Fonts
  3. index.html Add desktop V3 sidebar shell
  4. css/layout.css Rewrite for V3 sidebar on desktop. Mobile gains 4-tab bottom nav + new header icons.
  5. css/tracker.css and components, switch hardcoded colors to tokens
  6. assets/icons/favicon.svg Replace with red-on-ink favicon
  7. assets/icons/app-icon.svg Create for PWA
  8. assets/icons/mark.svg Create generic mark
  9. Settings panel New, four sections + Notifications stub
  10. Shop stub Placeholder route

Phased rollout

PhaseShip
1Side A only. Theme toggle shows Side A in both positions.
2Side B. Theme toggle works. User choice persists in Firestore.
3Settings panel with all four sections functional. Notifications "Coming soon."
4First paid theme (Rockstar). Shop becomes a functional panel.

Phases are independent. Each is a self-contained PR sequence.

What does NOT change

Firestore schema. Firestore rules. Auth flow. Cloud Functions. Privacy tiers. Diary paths. Log paths. Following arrays. All business logic is unaffected.
